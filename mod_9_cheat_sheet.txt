Prompt: Given `<!ELEMENT instructor (name, course+, officeHours*, phone | email)>`, which element must appear at least once?
Options:
  a. <course>
  b. <officeHours>
  c. <phone>
  d. <!ELEMENT>
Answer: <course> (a)
Explanation: The + quantifier makes <course> mandatory at least once.
---
Prompt: What does `<!ELEMENT instructor (name+, course?, officeHours*, phone | email)>` imply?
Options:
  a. The element name is optional.
  b. The element officeHours is optional.
  c. The element email is required.
  d. The element phone is required.
Answer: The element officeHours is optional. (b)
Explanation: The ? quantifier marks <course> as optional, whereas name+ requires at least one <name>.

=== TOPIC: XML Schema ===
Prompt: Why was XML Schema introduced after DTD? Select all that apply.
Options:
  a. DTDs are limited in the amount/type of data they can describe
  b. DTDs are not recommended in new XML scenarios
  c. DTDs cannot validate datatypes
  d. DTDs are too slow
  e. XML Schema follows XML syntax
Answer: DTDs are limited in the amount/type of data they can describe, DTDs cannot validate datatypes, XML Schema follows XML syntax (a, c, e)
Explanation: XML Schema adds strong typing and uses XML syntax itself, overcoming DTD limits.

=== TOPIC: XSLT expressivity ===
Prompt: What is the capacity of XSL as a programming language? Select all that apply.
Options:
  a. It has conditional capabilities comparable to OO languages
  b. It can express conditions using if-then-else
  c. It can express loops
  d. It is Turing complete and supports general computing
Answer: It has conditional capabilities comparable to OO languages, It can express conditions using if-then-else, It can express loops, It is Turing complete and supports general computing (a, b, c, d)
Explanation: XSLT can branch and iterate, making it computationally complete.
